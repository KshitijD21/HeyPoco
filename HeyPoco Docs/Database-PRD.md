# HeyPoco — Database Design PRD

**Version:** 1.2
**Date:** 2026-02-24
**Status:** Draft

---

## 1. Overview

HeyPoco's database stores unstructured life logs as raw text, auto-extracts lightweight metadata from them, and uses vector embeddings to power semantic querying. The schema is intentionally minimal — the intelligence is in the AI layer, not the database.

**Core Principle:** Store raw text faithfully. Extract metadata as a best-effort overlay. Never let extraction failure block storage.

---

## 2. Design Principles

- **Raw text is the source of truth** — `raw_text` is always stored exactly as spoken/typed
- **JSONB over columns** — extracted fields are flexible JSONB, not rigid columns. Unknown places like "Zigle's" and new field types don't require schema changes
- **Embeddings enable semantic search** — we use OpenAI `text-embedding-3-small` (1536 dims) on the `raw_text` at ingest time. This is what lets "coffee" queries surface "Starbucks" and "Zigle's" entries
- **Soft typing** — `entry_type` is a rough label (finance, event, etc.), not a hard constraint. The LLM reasons over embeddings at query time, not types
- **RLS at the database layer** — user isolation enforced in Postgres, not just application code

---

## 3. Existing Schema Assessment

### What to keep

| Component | Reason |
|-----------|--------|
| 2-table structure (`profiles` + `entries`) | Correct and minimal |
| `raw_text TEXT NOT NULL` | The most important column |
| `embedding VECTOR(1536)` | Powers all semantic queries |
| RLS policies | Correct — enforce user isolation at DB layer |
| GIN indexes on `tags` and `extracted_fields` | Needed for fast filtering |
| `handle_new_user()` trigger | Auto-creates profile on Supabase signup |
| `match_entries()` function | Good foundation, needs one improvement |

### What needs to change

| Issue | Current | Fix |
|-------|---------|-----|
| Rigid ENUM for `type` | `entry_type ENUM` — painful to change | Switch to `TEXT + CHECK` constraint — add new types with one line |
| No input source tracking | No `source` field | Add `source` column: `'voice'` or `'text'` |
| `extracted_fields` undocumented | Raw JSONB, no schema contract | Document + expand field set |
| `match_entries()` returns no similarity score | `SETOF entries` | Add `similarity FLOAT` to return type |
| No composite index | Separate indexes on `user_id` and `created_at` | Add `(user_id, created_at DESC)` composite |

### 4.2 `entries` — Core Log Table (Updated)

```sql
CREATE TABLE public.entries (
    id               UUID        PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id          UUID        NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
    type             TEXT        NOT NULL DEFAULT 'general'
                                 CHECK (type IN ('finance', 'event', 'general')),
    source           TEXT        NOT NULL DEFAULT 'text'
                                 CHECK (source IN ('voice', 'text')),
    raw_text         TEXT        NOT NULL,
    extracted_fields JSONB       NOT NULL DEFAULT '{}',
    tags             TEXT[]      NOT NULL DEFAULT '{}',
    attachments      TEXT[]      NOT NULL DEFAULT '{}',
    embedding        VECTOR(1536),
    created_at       TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    updated_at       TIMESTAMPTZ DEFAULT NOW() NOT NULL
);
```

**Adding a new type in future — one statement:**
```sql
ALTER TABLE entries DROP CONSTRAINT entries_type_check;
ALTER TABLE entries ADD CONSTRAINT entries_type_check
    CHECK (type IN ('finance', 'event', 'general', 'health'));
```

No type recreation. No data migration. No rename dance.

**Column rationale:**

| Column | Purpose |
|--------|---------|
| `raw_text` | Original user input, verbatim. Never modified after creation. |
| `source` | How the entry was created: `voice` (Whisper transcription) or `text` (typed). |
| `type` | Rough AI classification. Used for feed filtering only — not used for semantic queries. |
| `extracted_fields` | JSONB bag of structured metadata extracted by GPT-4o. Full schema documented below. |
| `tags` | 1–5 short labels generated by GPT-4o. e.g. `["coffee", "expense", "morning"]`. |
| `embedding` | 1536-dim vector of `raw_text`. Powers all semantic search. Nullable — entries without embeddings are excluded from vector search but still stored. |

### 4.3 `extracted_fields` — JSONB Contract

This column is JSONB — any new field can be added without a schema migration. GPT-4o extracts what it can; absent fields are simply omitted.

```jsonc
{
  // Finance (type = "finance")
  "amount":        15.00,                   // Numeric spend/income amount
  "currency":      "USD",                   // ISO 4217 code, default "USD"
  "merchant":      "Starbucks",             // Business name or person paid

  // Event (type = "event")
  "scheduled_at":  "2026-02-24T14:00:00",  // When something is planned to happen (ISO 8601)
  "duration_min":  120,                     // Duration in minutes ("2 hours" → 120)
  "location":      "friend's house",        // Where it happened / will happen
  "activity":      "lunch",                 // What the event is about

  // Any type — GPT-4o extracts these regardless of type
  "person":        "Rahul",                 // People mentioned
  "url":           "https://example.com",   // Links referenced
  "company":       "Acme Corp",             // Companies mentioned
  "role":          "Software Engineer",     // Roles / job titles
  "due_date":      "2026-03-01",            // Deadlines (ISO date)
  "notes":         "any freeform detail"    // Catch-all for anything else
}
```

> `link`, `career`, `document` are no longer entry types. Career or link entries will have `type = "general"` with the relevant fields extracted into JSONB and the right tags applied (e.g. `["career", "job-application"]`, `["link", "reference"]`). Tags handle the fine-grained classification — `type` is only for broad feed filtering.

**Why JSONB instead of columns?**
We can't predict what users will say. "I met Rahul for coffee" introduces a `person`. "Dentist appointment tomorrow at 10" introduces `scheduled_at`. Separate columns would require a migration for each new concept. JSONB absorbs new fields without any schema change, and GIN indexing keeps queries efficient.

## 5. `match_entries()` — Vector Search Function (Updated)

The current function returns `SETOF entries` with no similarity score. The updated version exposes the score so the query layer can distinguish strong matches from loose "maybe related" results.

```sql
-- Return type: entry columns + cosine distance score
-- Distance: 0.0 = identical, 2.0 = opposite
CREATE TYPE entry_match AS (
    id               UUID,
    user_id          UUID,
    type             TEXT,
    source           TEXT,
    raw_text         TEXT,
    extracted_fields JSONB,
    tags             TEXT[],
    attachments      TEXT[],
    embedding        VECTOR(1536),
    created_at       TIMESTAMPTZ,
    updated_at       TIMESTAMPTZ,
    similarity       FLOAT
);

CREATE OR REPLACE FUNCTION match_entries(
    query_embedding      VECTOR(1536),
    match_user_id        UUID,
    match_count          INT   DEFAULT 10,
    similarity_threshold FLOAT DEFAULT 1.0  -- 1.0 = no filter
)
RETURNS SETOF entry_match
LANGUAGE sql
STABLE
AS $$
    SELECT
        e.id, e.user_id, e.type, e.source,
        e.raw_text, e.extracted_fields, e.tags, e.attachments,
        e.embedding, e.created_at, e.updated_at,
        (e.embedding <=> query_embedding) AS similarity
    FROM public.entries e
    WHERE e.user_id = match_user_id
      AND e.embedding IS NOT NULL
      AND (e.embedding <=> query_embedding) <= similarity_threshold
    ORDER BY e.embedding <=> query_embedding
    LIMIT match_count;
$$;
```

**Similarity score thresholds (cosine distance):**

| Range | Meaning | LLM behaviour |
|-------|---------|---------------|
| `< 0.3` | Strong match | Answer directly from entry |
| `0.3 – 0.6` | Related match | Cite as "possibly related" |
| `> 0.6` | Weak match | Exclude or surface with low confidence |

This is what enables graceful degradation: when a user asks "how much did I spend on coffee?" and only "Zigle's" is logged, the score tells the LLM whether to treat it as a confident answer or a tentative suggestion.

## 6. Indexes

```sql
-- Existing (keep as-is)
CREATE INDEX idx_entries_user_id           ON public.entries(user_id);
CREATE INDEX idx_entries_type              ON public.entries(type);
CREATE INDEX idx_entries_created_at        ON public.entries(created_at DESC);
CREATE INDEX idx_entries_tags              ON public.entries USING GIN(tags);
CREATE INDEX idx_entries_extracted_fields  ON public.entries USING GIN(extracted_fields);

-- New: composite index for the dominant query pattern
CREATE INDEX idx_entries_user_created      ON public.entries(user_id, created_at DESC);
```

**Why the composite index?**
Almost every feed query is `WHERE user_id = ? ORDER BY created_at DESC`. The composite index covers the filter and sort in a single scan, eliminating a separate sort step.

## 7. RLS Policies — No Changes

The existing policies are correct. All four operations are user-scoped:

```sql
-- SELECT, INSERT, UPDATE, DELETE all enforce: auth.uid() = user_id
```

> Note: The backend uses the Supabase **service role key**, which bypasses RLS. RLS is a safety net for any future direct client access (mobile app, third-party integrations, etc.).

## 8. Data Flow

```
User says: "I went to Zigle's today and spent $20"
                        ↓
              raw_text stored verbatim
                        ↓
   GPT-4o extraction (best-effort, non-blocking):
   { type: "finance", extracted_fields: { amount: 20, merchant: "Zigle's" }, tags: ["expense"] }
                        ↓
   OpenAI embedding generated from raw_text → stored in embedding column
                        ↓
User asks: "How much did I spend on coffee?"
                        ↓
   Question embedded → vector search → top results returned with similarity scores:
     - Starbucks $15  (similarity: 0.18 — strong match)
     - Zigle's $20    (similarity: 0.34 — related match)
                        ↓
   LLM reasons over results + scores:
   "I found Starbucks ($15), which is a coffee shop.
    I also found Zigle's ($20) — I'm not certain it's a coffee place,
    but it may be related. Confirmed coffee spend: $15. Possibly up to $35."
```

## 10. Migration Plan

Builds incrementally on the existing 4 migrations.

| Migration | Change |
|-----------|--------|
| `005_drop_enum_add_text_type_and_source.sql` | Drop `entry_type` ENUM, change `type` column to `TEXT + CHECK (finance, event, general)`, add `source` column |
| `006_update_match_entries.sql` | Create `entry_match` type + replace `match_entries()` with similarity-aware version |
| `007_add_composite_index.sql` | `CREATE INDEX idx_entries_user_created ON entries(user_id, created_at DESC)` |

> **To add a new type later:** just drop and re-add the CHECK constraint with the new value included. No data touched, no type recreation needed.
